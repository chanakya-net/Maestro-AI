Introduction

AI-Skills is a reusable skill collection for coding agents — Copilot, Codex, Claude, Gemini, and others. It provides a practical workflow from requirements discovery → issue planning → orchestrated execution, with shared runtime assets so behavior stays consistent across agents and operating systems.

The collection is designed around a pipeline architecture: each skill handles one stage of the software delivery lifecycle, and they chain together naturally. The crown jewel is run-with-it, a two-layer orchestration runtime that can drive multi-issue execution for hours without context degradation.

Run With It — Architecture

run-with-it is a two-layer orchestration runtime designed for indefinite multi-issue execution without context degradation. It separates concerns between a persistent Main Orchestrator and ephemeral Sub-Coordinators.

Main Orchestrator

Runs in the primary agent session. Fetches issues, plans execution order, spawns one Sub-Coordinator per issue, reads compact reports, updates GitHub. Never implements code. Memory is .run-with-it/main-state.json.

Sub-Coordinator

Spawned per issue via run-agent.sh. Handles complexity analysis, agent routing, implementation, review, and modification. Writes a compact report JSON when done. Never touches GitHub. Each runs with MAX_AGENT_DEPTH=1.

Existing-State Detection

If .run-with-it/main-state.json exists, prompt user: resume (continue previous run) or discard (delete and start fresh). On resume, in_progress issues reset to pending (Sub-Coordinators are ephemeral).

2. Initial Issue Fetch & Execution Plan

After preflight clears, issues are fetched and sequenced:

  1. Primary: gh issue list --label ready-for-agent --state open --limit 10. Retry outside sandbox if sandbox fails.
  2. Fallback: Local issues.md file — only when gh fails both sandboxed and unsandboxed, or no GitHub remote exists.
  3. Dependency graph: Each issue’s body/labels/cross-references are parsed to identify required dependencies.
  4. Topological sort: Respects dependency order. Within same tier: critical fixes → infrastructure → feature slices → polish → refactors.
  5. Blocked detection: Issues whose deps are open/unresolved get blocked status until those deps complete.

3. Main Orchestrator Loop

Executes immediately and unconditionally after the plan is written — never pauses for user input after planning. Each iteration:

StepActionKey Details
A Memory Refresh Re-read main-orchestrator-rules.md + main-state.json. Mandatory at top of every iteration. Emit STATUS memory-refresh line.
B Identify Next Issue First pending issue whose deps are all completed. Re-evaluate blocked issues. Exit loop when all terminal.
C Assemble Context gh issue view <n> (fresh), recent commits, CodeGraph/grep context, env block with SUB_COORD_ISSUE_NUMBER, SUB_COORD_REPORT_FILE, MAX_AGENT_DEPTH=1, all overrides. Mark issue in_progress in state.
D Spawn Sub-Coordinator run-agent.sh --prompt-file sub-coordinator-prompt.md with dangerouslyDisableSandbox: true. Background + monitored loop. 3600s stall alert. Shell watcher prints only changed status lines. Never load logs into AI context.
E Collect Report Read .run-with-it/reports/sub-<n>-report.json. If missing: blocked + report-missing. Delete context file. Never read log files.
F Update State + GitHub Write main-state.json. Post terminal comment using Appendix E markdown template (Status/Summary/Verification/Token Usage/Notes/Blocking Reasons). Close if completed; leave open if failed-review or blocked.

Critical Guardrails

  • Never implement work directly in this session
  • Never run tests, build commands, or compile
  • Never load sub-coordinator log files into AI context
  • Never kill, cancel, or restart a Sub-Coordinator mid-run
  • Never derive issue state from conversation history — always read main-state.json
  • Never inspect or infer Sub-Coordinator internal routing decisions
  • Never present execution option menus or pause after planning

4. Sub-Coordinator Internals

Each Sub-Coordinator is a self-contained agent session that handles exactly one issue end-to-end. It is spawned via run-agent.sh with MAX_AGENT_DEPTH=1 (children cannot spawn further sub-agents). It never touches GitHub and never updates main-state.json.

1 Complexity Analysis
Spawn complexity sub-agent using complexity-prompt.md. 9-dimension rubric, scores 1-5 each (total 9-45). Maps to band: quite-easy, easy, medium, medium-hard, complex, holy-fuck. Output: COMPLEXITY| line + JSON blob.
↓
2 Deterministic Routing
Score to weight range. Build candidate pool from agent-registry.json. Sort by complexity_weight ASC, price_tier ASC, price_output_per_1m ASC. Randomly pick from top 4. Select agent that supports the chosen model. Apply allowlist/denylist/provider routing rules.
↓
3 Implementation
Spawn implementer worker-agent with prompt.md. Runs TDD discipline (red-green-refactor). Produces code with behaviour-first tests through public interfaces. Captures commit SHA, not diff text.
↓
Gate Skip review if < 30 lines, ≤ 3 files, all tests pass
↓
4 Review
Reviewer band = implementer band + 1. Reviewer writes status JSON: verdict approve/revise/reject + instructions. Sub-Coordinator reads ONLY the status file (verdict + comment_count + nitpick_only).
↓
approve → integrate & commit
revise → modification loop (max 4 cycles, escalate band after 2)
reject → terminate as failed-review
↓

5. Cleanup

Successful Run

Delete all .run-with-it/ files, remove untracked technical_requirements.md/prd.md/issues.md, append to .gitignore, commit with chore: remove skill-generated artifacts post-run.

Failed / Interrupted

Preserve all .run-with-it/ files. Print paths of preserved state, reports, logs, reviews. Offer discard command to force-delete and restart.

Discard

Delete everything including main-state.json, all reports, logs, reviews, technical_requirements.md, prd.md, issues.md. Update .gitignore, commit. Proceed as fresh run.

Safety: Before deleting any workspace file, check git status --short. Never delete user-modified files. Skip dirty files and emit STATUS|type=cleanup|action=skipped-dirty-file.

Resume & Compression Survival

Resume Flow

On startup, if main-state.json exists, prompt resume/discard. On resume: re-read state, reset in_progress to pending (Sub-Coordinators are ephemeral), skip already-terminal ones, re-enter Main Loop at Step A.

Compression Survival

After context compression: re-read main-state.json. completed issues are never re-run. in_progress resets to pending. completed_summaries gives bounded history. Never derive state from conversation.

ephemeral main-state.json completed_summaries re-read before each loop
